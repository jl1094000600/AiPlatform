# AI Platform README 优化设计

日期：2026-06-11

## 1. 目标

重写仓库根目录 `README.md`，修复现有中文乱码和内容过时问题，并在 GitHub 仓库首页清晰展示 AI Platform 当前的产品范围、工程能力、技术架构与项目状态。

README 面向开源开发者、招聘方、技术同行和潜在平台用户，以“技术产品展示”为主。内容使用中文，关键技术术语保留英文。

## 2. 对外定位

主标题使用 `AI Platform`。

核心定位：

> 面向 AI 应用全生命周期的一体化工程平台。

辅助说明应明确平台统一覆盖 Agent、RAG、PromptOps、评测治理、自动化交付、多租户权限和成本观测，避免继续将项目描述为早期的单一 Agent 管理后台。

项目成熟度统一表达为：

> 功能主体完成，持续迭代与环境联调中。

不得使用“生产就绪”“企业生产级”等未经完整环境验证的表述。

## 3. 信息架构

README 按以下顺序组织：

1. 首屏产品介绍与关键界面
2. 项目定位、适用场景与产品边界
3. 核心能力矩阵
4. 更多产品界面
5. 系统架构
6. 技术栈
7. 项目结构
8. 快速启动
9. 当前状态与路线图
10. 文档导航
11. 参与贡献与 License

首屏的主要目标是展示产品完成度，引导访客浏览界面截图、阅读架构文档并关注项目。两张关键界面作为首屏内容的一部分，用产品证据承接定位；其余截图在后文按能力补充。快速启动放在正文中段，不抢占第一视觉层级。

## 4. 首屏设计

首屏采用 GitHub Markdown 原生能力完成，保持稳定、轻量和跨主题可读性。

首屏包含：

- 居中的项目名称和一句话定位。
- 一段不超过两行的能力概述。
- 5 至 7 个高价值徽章，建议覆盖 Java、Spring Boot、Vue、License 和项目状态。
- 一个定制品牌横幅，包含 AI Platform 名称、平台定位和克制的工程视觉元素。
- 两张最能证明平台完成度的真实界面截图，优先选择仪表盘和 Agent 调用图谱。
- 简短的页内导航：功能概览、产品界面、系统架构、快速启动、项目文档。

避免使用动态访客统计、大量无意义徽章、复杂 HTML 布局和依赖不稳定的外部图片服务。

品牌横幅不照搬其他项目的构图、色彩或文案。视觉应体现 AI 工程平台的可靠、清晰和可观测性，避免过度装饰、霓虹堆叠、概念化机器人图像或与实际产品无关的生成式插画。

## 5. 产品定位与边界

在截图之后增加简短的“这是什么”与“它不是什么”，帮助不同受众快速建立正确预期。

### 5.1 这是什么

- 面向团队的 AI 应用工程与治理平台。
- 将 Agent、知识、提示词、评测、交付、权限和成本放进统一工作流。
- 适合研究 AI 平台工程、构建内部 AI 能力底座或演示完整 AI DevOps 流程。

### 5.2 它不是什么

- 不是只提供对话页面的 Chat UI。
- 不是只有 Agent 增删改查的管理后台。
- 不是已经完成所有生产环境验证的商业云服务。

边界表达应保持简短和专业，不采用贬低其他项目类型的语气。

## 6. 核心能力

能力矩阵分为七个领域。每个领域先说明解决的问题或形成的工作流，再列出具体能力，避免只罗列名词。

### 6.1 Agent 生命周期与协作

- Agent 注册、心跳、上下线和版本管理。
- A2A 通信、调用链路与关系图谱。
- 工作流编排和执行记录。

### 6.2 RAG 知识库

- Chroma 向量存储和文档块查看。
- Embedding 模型选择。
- 固定切分主路径，以及持续完善中的混合切分能力。

### 6.3 PromptOps 与 Skill

- 提示词版本、评测、发布和 Agent 运行时关联。
- Skill 创建、AI 辅助生成和流水线上下文注入。

### 6.4 评测与 AI 治理

- 数据集导入、模拟数据生成和 Agent 测评。
- 代码质量评估、质量门禁、人工审核和策略治理。

### 6.5 自动化交付

- PRD 生成、代码生成、产物预览和重新生成。
- 构建、测试、Docker/Jenkins 部署配置和运行记录。

### 6.6 多租户与权限

- 租户切换、成员、角色、权限码和菜单树。
- JWT 租户上下文、RBAC 校验、数据隔离和审计。

### 6.7 模型与运营观测

- 模型目录、调用监控和 Token 统计。
- 用户维度成本计费、用户记忆和模型训练入口。

## 7. 产品界面

README 优先展示 3 至 4 张真实界面截图。前两张紧跟首屏并排展示，其余截图放在对应能力说明附近。建议选择：

1. 首页仪表盘或平台工作台。
2. Agent 调用关系图谱。
3. 自动化流水线详情。
4. PromptOps、RAG 或多租户权限页面中的一项。

截图统一放入 `docs/assets/readme/`，使用仓库相对路径引用。图片需要裁去无关浏览器边框和敏感数据，保证中文清晰，并在浅色、深色 GitHub 主题下都能看懂。

若当前缺少适合公开的截图，实施阶段通过本地运行平台采集真实界面。最终 README 不保留空白截图占位，也不使用与实际产品不符的概念图冒充截图。

## 8. 系统架构

架构图使用 GitHub 原生 Mermaid，表达以下关系：

- Vue 3 管理端和 Consumer Front。
- Spring Boot API 与核心平台服务。
- Agent、Workflow、RAG、PromptOps、Evaluation、Governance、Tenant/RBAC 等领域模块。
- MySQL、Redis、Chroma 和 Embedding Service。
- 外部模型提供方与独立 Agent 服务。
- Docker/Jenkins 自动化交付目标。

架构图保持一屏可理解，不展开数据库表或类级实现细节。详细设计通过文档导航链接到 `docs/`。

架构图之后增加简短的技术选型说明，解释 Spring Boot、Vue、MySQL/Redis、Chroma 和独立 Agent 服务在当前平台中的职责。说明以项目实际架构为依据，不写通用框架宣传语。

## 9. 技术与启动信息

技术栈以当前代码为准：

- Backend：Java 21、Spring Boot 3.2.4、Spring AI、MyBatis-Plus、Spring Security、JWT。
- Frontend：Vue 3、Vite 5、Element Plus、Pinia、ECharts、Axios。
- Data & Infra：MySQL 8、Redis、Chroma、Docker、Jenkins。
- AI Services：Embedding Service、BGE-M3 Training、Image Agent、Marketing Agent、Intent Agent。

快速启动需区分：

- 推荐方式：仓库已有的统一启动脚本。
- 手动方式：基础依赖、后端、前端和可选 AI 服务分别启动。
- 环境要求：JDK 21、Maven、Node.js、MySQL、Redis；Chroma 和 Embedding Service 按使用场景启用。

README 优先展示统一启动脚本和最短可行路径。较长的手动配置、环境变量和生产部署步骤使用折叠区或链接到部署文档，避免快速启动被配置细节淹没。

不得在 README 中公开默认密码。示例账号如确有必要，应要求用户通过环境变量或初始化配置自行设置。

## 10. 状态与诚实表达

功能状态统一使用三类：

- 已完成：代码主体和基础验证已完成。
- 持续完善：主流程可用，仍有增强项。
- 待环境验证：依赖 Java/Maven、Redis、Docker、Jenkins、GPU 等真实环境的端到端验证。

当前应明确：

- 前端生产构建已有通过记录。
- 后端完整测试受当前环境缺少 Maven/JDK 影响，README 不宣称全部测试通过。
- Docker/Jenkins、Redis 记忆、Chroma 入库和 BGE-M3 训练仍需真实环境联调。
- RAG 混合切分处于持续完善状态，固定切分为当前主路径。

状态区域使用紧凑表格或任务清单呈现，并配一段短路线图。路线图只列已经存在明确代码方向或文档依据的事项，不加入未经规划的宏大功能。

## 11. 文档导航

README 只保留高价值入口，避免复制长篇设计内容。至少链接到：

- 产品需求文档。
- 系统架构文档。
- Ubuntu 部署文档。
- 开发规范。
- 测试或集成报告。
- 最新平台开发进度。

链接必须使用仓库相对路径，并在实施时逐项检查目标文件存在且锚点可用。

## 12. 文案与视觉规则

- 清晰优先，不使用“赋能”“领先”“革命性”等空泛营销词。
- 描述真实结果和工作流，不只列技术名词。
- 中文与英文、数字之间保留合理空格。
- 每段控制在 2 至 4 行，长内容拆成表格或列表。
- 表格只用于能力矩阵、技术栈和状态信息，不把整个 README 做成表格集合。
- 图标可适量使用，但不依赖 emoji 表达唯一含义。
- 目录树只展示一级模块和关键服务，避免复制完整源码目录。
- 删除过时 API 清单；API 细节应进入独立文档。
- 首尾可自然加入 GitHub Star 引导，但不使用弹窗式、重复式或夸张的营销文案。
- 借鉴优秀 README 的信息节奏和证据优先原则，不复制其品牌图、章节文案、配色、图标或布局细节。
- README 的视觉重点必须来自 AI Platform 的真实界面、真实架构和真实能力，而不是装饰性素材。

## 13. 验收标准

- `README.md` 使用 UTF-8 编码，中文无乱码。
- GitHub 首屏在桌面端能看到项目名、定位、徽章和真实产品视觉。
- 访客在两分钟内能理解平台是什么、具备什么能力、当前成熟度如何。
- 所有功能描述均可在代码、提交记录或现有文档中找到依据。
- Mermaid 在 GitHub Markdown 中语法有效。
- 图片和文档链接使用相对路径且目标存在。
- 快速启动命令与仓库实际脚本和目录一致。
- 不公开默认密码、Token、连接凭据或本地敏感配置。
- 不宣称尚未完成的测试、联调或生产能力。
- README 在浅色和深色 GitHub 主题下均保持可读。
- 首屏和前两个章节形成“定位 -> 真实界面 -> 产品边界”的清晰阅读链路。
- 品牌横幅、截图和架构图均体现 AI Platform 自身内容，不构成对参考项目的视觉复刻。
