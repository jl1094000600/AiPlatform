# Java 代码生成与编写规范

为保证代码的可读性、可维护性与高内聚低耦合特性，特制定以下 Java 代码生成规范。

## 1. 命名规范 (Naming Conventions)

代码命名需做到**见名知意**，严禁使用拼音与英文混合，严禁使用无意义的单字母（除局部循环变量外）。

* **包名 (Package)**：统一使用小写，点分隔符之间有且仅有一个自然语义的英语单词。包名统一使用单数形式（例如 `com.company.project.util` 而不是 `utils`）。
* **类与接口 (Class & Interface)**：统一使用 **UpperCamelCase**（大驼峰）风格。
* *示例*：`UserAccountService`, `OrderProcessor`。
* *特例*：DO/BO/DTO/VO/AO 等领域模型缩写需全部大写，如 `UserDTO`。


* **方法 (Method)**：统一使用 **lowerCamelCase**（小驼峰）风格。动词开头，描述具体行为。
* *示例*：`getUserById`, `calculateTotalPrice`, `insertOrder`。


* **变量 (Variable)**：统一使用 **lowerCamelCase**（小驼峰）风格。
* 布尔类型变量严禁加 `is` 前缀，避免部分 RPC 框架或序列化工具解析引起错误（例如用 `deleted` 而不是 `isDeleted`）。


* **常量 (Constant)**：统一使用 **UPPER_SNAKE_CASE**（全大写加下划线）风格，力求语义表达完整。
* *示例*：`MAX_RETRY_COUNT`, `DEFAULT_PAGE_SIZE`。



## 2. 方法闭环与健壮性 (Method Closure)

方法必须形成完整的逻辑闭环，保证在任何输入和异常状态下都有可预期的结果，避免引发不可控的连锁错误。

* **参数校验 (Fail-Fast)**：在方法入口处，必须对入参进行合法性校验（判空、边界值检查等）。不合法的参数应立即抛出异常或返回错误码，避免无效逻辑继续向下执行。
* **单一职责**：核心逻辑必须保持单一职责，只做方法签名所描述的事情，不产生意料之外的副作用。
* **异常处理闭环**：
* 能够预见并可降级恢复的异常，必须在方法内部 `try-catch` 并进行妥善处理。
* 无法处理的系统级异常需向上抛出，交由顶层或全局异常处理器统一接管。
* 严禁吞异常（`catch` 块为空，或仅打印日志却不中断程序）。


* **资源释放闭环**：涉及文件、流、数据库连接等外部资源的操作，必须在 `finally` 块中明确释放，或强制使用 `try-with-resources` 语法。
* **出口清晰**：尽量减少方法内部的提前 `return`，或者保证所有分支的返回值结构统一，逻辑链路有始有终。

## 3. 方法长度与复杂度控制

为保证代码逻辑清晰且易于 Code Review，对方法体长度做严格限制。

* **行数红线**：单个方法的总行数**严禁超过 120 行**（包含代码、注释和空行）。
* **建议标准**：优秀的方法应控制在 **80 行以内**，确保一屏刚好能看完整个方法的逻辑，无需频繁滚动。
* **重构触发点**：
* 当方法逼近或超过 120 行时，强制要求进行方法拆分。
* 代码嵌套层级（`if / for / while / switch`）超过 3 层时，必须将内层逻辑抽离为独立的方法。
* 庞大的 `switch-case` 或 `if-else` 链，强制要求使用多态、策略模式或枚举来进行重构。



## 4. 公共方法提取与重用 (DRY 原则)

坚决杜绝“复制粘贴”式的代码复用（Don't Repeat Yourself）。当一段逻辑在代码中出现两次及以上时，必须考虑提取。

* **工具类 (Utils / Helpers)**：
* 没有任何状态依赖、纯粹的输入输出逻辑，必须提取到公共工具类中（如 `DateUtils`, `StringUtil`）。
* 工具类必须将构造方法私有化（`private`），防止被无意义实例化。
* 工具类的方法必须声明为 `public static`。


* **业务逻辑复用**：
* **类内复用**：在同一类中重复出现的逻辑片段，应提取为 `private` 私有方法，通过良好的命名提升主流程的自解释性。
* **跨类复用**：跨模块或跨类重复使用的业务逻辑，应提取到统一的公共服务类（Common Service）或定义在父类抽象类（Base Class）中。


* **上下文封装**：当提取的公共方法入参超过 3 个时，必须将参数封装为一个 Context 对象或 DTO 实体进行传递，以提升接口的稳定性和扩展性。
